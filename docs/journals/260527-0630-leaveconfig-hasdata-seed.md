# LeaveConfig HasData Seed -- Migrations-First Baseline

**Date**: 2026-05-27 06:30
**Severity**: Medium
**Component**: LeaveConfig seed data + MigrateLegacyStatusesAsync
**Status**: Resolved

## What Happened

`MigrateLegacyStatusesAsync` relies on `LeaveConfig` rows to calculate `max approval levels` for `approved` status rows. Without seed data, fresh databases would fall back to `ApprovedLevel=1` for all approved requests -- incorrect for 2-level leave types (NPN, NO, NVR, NTS). Added `HasData` block in `AppDbContext.OnModelCreating` with 9 rows covering the initial approval configuration baseline, then generated migration `SeedLeaveConfigs`.

## The Brutal Truth

This is a data-availability-at-startup problem that should have been obvious from the moment `MigrateLegacyStatusesAsync` was designed. The migration logic queries `LeaveConfig` to determine approval depth, but nobody ensured the data existed before that query ran. If we had tested against a truly fresh database (not one with manually inserted config rows), this would have been caught immediately. The fact that we needed a separate seed migration after the entity migration proves the entity migration was incomplete on its own.

## Technical Details

- **AppDbContext.cs**: Added `modelBuilder.Entity<LeaveConfig>().HasData(...)` block inside `OnModelCreating` with 9 rows: NPN 2-level, NO 2-level, NVR 2-level, NKL 1-level, NTS 2-level (each with level-1 and level-2 entries, plus NKL with level-1 only).
- **Migration `20260527061558_SeedLeaveConfigs`**: Auto-generated by `dotnet ef migrations add SeedLeaveConfigs`. Inserts the 9 rows via INSERT statements, ensuring data exists before `MigrateLegacyStatusesAsync` runs at startup.
- **Comment added**: Clarifying note that `HasData` seed is baseline-only; runtime updates via `Config/Update` endpoint (`ReplaceAllAsync`) replace these rows.

## What We Tried

- **Option A (HasData in AppDbContext)**: Chosen. Data flows through EF migration pipeline, guaranteed present before `MigrateLegacyStatusesAsync` executes. Downside: migration regeneration needed if seed values change, but baseline values are stable.
- **Option B (SeedHelper upsert)**: Rejected. `SeedHelper` runs in `Program.cs` after `MigrateAsync`, but `MigrateLegacyStatusesAsync` also runs at startup -- no guaranteed ordering. Race condition risk was unacceptable for a one-time data migration that depends on config data being present.

## Root Cause Analysis

The entity migration (`AddLeaveConfigTable`) created the schema but left the table empty. `MigrateLegacyStatusesAsync` was written assuming config data would be available, but the only insertion path was the `Config/Update` endpoint (runtime, user-triggered). Fresh databases had zero config rows, causing all 2-level leave types to default to `ApprovedLevel=1`. The seed data gap between "schema exists" and "data needed for migration logic" was not identified during the phase-01 design.

## Lessons Learned

1. **Any startup migration that queries a table must ensure that table has seed data before it runs.** This is not optional -- it is a hard dependency.
2. **HasData is the correct EF Core pattern for baseline data that must exist before application logic runs.** SeedHelper is for runtime-upsertable data where ordering does not matter.
3. **Test migration logic against a truly empty database**, not one warmed up by manual config inserts. The bug was invisible in dev because config rows were manually created during feature development.
4. **Document data dependencies between migrations.** A comment like "MigrateLegacyStatusesAsync requires LeaveConfig rows" should have been in the migration file from the start.

## Next Steps

- Verify `MigrateLegacyStatusesAsync` produces correct `ApprovedLevel` values against a fresh database (all leave types, both 1-level and 2-level).
- Confirm `Config/Update` endpoint `ReplaceAllAsync` correctly overrides seed data at runtime without migration conflicts.
- Close out phase-01 entity-migration checklist items.
